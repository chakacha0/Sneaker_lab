# Блок-схема алгоритма авторизации пользователя

## Алгоритм: Авторизация пользователя (User Login/Authentication)

### Описание
Алгоритм описывает процесс авторизации пользователя в системе: от ввода email и пароля до сохранения данных пользователя в localStorage и обновления интерфейса.

## Текстовое описание алгоритма

### Шаг 1: Начало
Пользователь открывает модальное окно авторизации и вводит данные

### Шаг 2: Ввод данных
- Пользователь вводит `email` и `password` в форму
- Нажимает кнопку "Login"

### Шаг 3: Инициализация процесса
- Установка `setIsLoading(true)` - флаг загрузки
- Очистка сообщений: `setMessage("")`

### Шаг 4: Отправка HTTP запроса
- POST запрос на `/users/login`
- Данные: `{email, password}`

### Шаг 5: Валидация на сервере
- Проверка наличия обязательных полей: `email` и `password`
- Если данные невалидны → HTTPException 400 "Email и пароль обязательны" → вывод ошибки → КОНЕЦ

### Шаг 6: Поиск пользователя в базе данных
- Выполнение SQL запроса:
  ```sql
  SELECT user_id, role, email, password_hash, 
         first_name, last_name, phone_number, email_verified
  FROM users
  WHERE email = %s
  ```

### Шаг 7: Проверка существования пользователя
- Если пользователь не найден → return None → HTTPException 401 "Неверный email или пароль" → вывод ошибки → КОНЕЦ

### Шаг 8: Проверка пароля
- Получение `password_hash` из базы данных
- Вызов `bcrypt.checkpw(password.encode("utf-8"), password_hash.encode("utf-8"))`
- Сравнение введенного пароля с хэшем из базы данных

### Шаг 9: Валидация пароля
- Если пароль неверный → return None → HTTPException 401 "Неверный email или пароль" → вывод ошибки → КОНЕЦ
- Если пароль верный → продолжение

### Шаг 10: Удаление чувствительных данных
- Удаление `password_hash` из объекта пользователя (не передаем хэш клиенту)

### Шаг 11: Проверка подтверждения email
- Проверка: `user.email_verified == True?`
- Если email не подтвержден → HTTPException 403 "Email не подтвержден. Пожалуйста, подтвердите email перед входом" → вывод ошибки → КОНЕЦ

### Шаг 12: Формирование ответа
- Создание объекта ответа:
  ```json
  {
    "message": "Вход выполнен успешно",
    "user": {
      "user_id": ...,
      "email": ...,
      "first_name": ...,
      "last_name": ...,
      "phone_number": ...,
      "role": ...,
      "email_verified": true
    }
  }
  ```

### Шаг 13: Обработка результата на клиенте
- **Если успешно:**
  - Вывод сообщения: `setMessage(result.message)`
  - Сохранение пользователя в localStorage: `localStorage.setItem("user", JSON.stringify(result.user))`
  - Ожидание 1 секунды
  - Закрытие модального окна: `onClose()`
  - Обновление страницы: `window.location.reload()`
- **Если ошибка:**
  - Вывод сообщения об ошибке: `setMessage(error.message)`

### Шаг 14: Завершение
- Сброс флага загрузки: `setIsLoading(false)`
- КОНЕЦ

## Сложность алгоритма

- **Временная сложность:** O(1) - константная, так как поиск по email выполняется по индексу, а проверка пароля - константная операция
- **Пространственная сложность:** O(1) - константная память

## Особенности алгоритма

1. **Безопасность паролей:** Пароли хранятся в виде хэшей (bcrypt), никогда не передаются в открытом виде
2. **Проверка email:** Требуется подтверждение email перед авторизацией
3. **Валидация данных:** Проверка наличия обязательных полей на сервере
4. **Обработка ошибок:** Различные типы ошибок (400, 401, 403) для разных ситуаций
5. **Сохранение сессии:** Данные пользователя сохраняются в localStorage для поддержания сессии
6. **Обновление интерфейса:** Автоматическое обновление страницы после успешной авторизации

## Типы ошибок

### HTTP 400 (Bad Request)
- **Причина:** Не заполнены обязательные поля (email или password)
- **Сообщение:** "Email и пароль обязательны"

### HTTP 401 (Unauthorized)
- **Причина:** Неверный email или пароль
- **Сообщение:** "Неверный email или пароль"

### HTTP 403 (Forbidden)
- **Причина:** Email не подтвержден
- **Сообщение:** "Email не подтвержден. Пожалуйста, подтвердите email перед входом"

## Примеры сценариев

### Сценарий 1: Успешная авторизация
- Email: `user@example.com`
- Password: `correct_password`
- Email подтвержден: `true`
- **Результат:** Пользователь авторизован, данные сохранены в localStorage, страница обновлена

### Сценарий 2: Неверный пароль
- Email: `user@example.com`
- Password: `wrong_password`
- **Результат:** Ошибка 401 "Неверный email или пароль"

### Сценарий 3: Пользователь не найден
- Email: `nonexistent@example.com`
- Password: `any_password`
- **Результат:** Ошибка 401 "Неверный email или пароль"

### Сценарий 4: Email не подтвержден
- Email: `user@example.com`
- Password: `correct_password`
- Email подтвержден: `false`
- **Результат:** Ошибка 403 "Email не подтвержден. Пожалуйста, подтвердите email перед входом"

### Сценарий 5: Пустые поля
- Email: `""`
- Password: `""`
- **Результат:** Ошибка 400 "Email и пароль обязательны"

## Безопасность

1. **Хэширование паролей:** Используется bcrypt для безопасного хранения паролей
2. **Защита от перебора:** bcrypt делает подбор паролей вычислительно дорогим
3. **Не передача хэша:** password_hash удаляется из ответа перед отправкой клиенту
4. **Валидация на сервере:** Все проверки выполняются на сервере, клиентские проверки - только для UX
5. **Подтверждение email:** Дополнительный уровень безопасности через подтверждение email
