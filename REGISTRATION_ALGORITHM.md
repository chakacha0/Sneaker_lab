# Блок-схема алгоритма регистрации пользователя

## Алгоритм: Регистрация пользователя (User Registration)

### Описание
Алгоритм описывает процесс регистрации нового пользователя в системе: от заполнения формы регистрации до создания учетной записи, отправки кода подтверждения email и перехода на страницу подтверждения.

## Текстовое описание алгоритма

### Шаг 1: Начало
Пользователь открывает форму регистрации и вводит данные

### Шаг 2: Ввод данных
- Пользователь заполняет форму:
  - `email` (обязательно)
  - `password` (обязательно)
  - `first_name` (опционально)
  - `last_name` (опционально)
  - `phone_number` (опционально)
- Нажимает кнопку "Register"

### Шаг 3: Инициализация процесса
- Установка `setIsLoading(true)` - флаг загрузки
- Очистка сообщений: `setMessage("")`

### Шаг 4: Отправка HTTP запроса
- POST запрос на `/users/register`
- Данные: `{email, password, first_name, last_name, phone_number}`

### Шаг 5: Валидация на сервере
- Проверка наличия обязательных полей: `email` и `password`
- Если данные невалидны → HTTPException 400 "Email и пароль обязательны" → вывод ошибки → КОНЕЦ

### Шаг 6: Хэширование пароля
- Генерация соли: `bcrypt.gensalt()`
- Хэширование пароля: `bcrypt.hashpw(password.encode("utf-8"), salt)`
- Получение `password_hash`

### Шаг 7: Генерация кода подтверждения
- Вызов `generate_verification_code()`
- Генерация случайного 6-значного числа (от 100000 до 999999)
- Получение `verification_code`

### Шаг 8: Создание пользователя в базе данных
- Выполнение SQL запроса:
  ```sql
  INSERT INTO users (role, email, password_hash, first_name, 
                    last_name, phone_number, email_verified, verification_code)
  VALUES ('customer', %s, %s, %s, %s, %s, FALSE, %s)
  RETURNING user_id, role, email, first_name, last_name, 
            phone_number, email_verified, verification_code
  ```

### Шаг 9: Обработка результата создания
- **Если успешно:**
  - Получение данных созданного пользователя
  - COMMIT транзакции
  - Продолжение
- **Если ошибка:**
  - ROLLBACK транзакции
  - HTTPException 400 с описанием ошибки (например, "Email уже используется")
  - Вывод ошибки на клиенте → КОНЕЦ

### Шаг 10: Получение кода подтверждения
- Извлечение `verification_code` из объекта пользователя
- Если код не сгенерирован → `email_sent = False` → переход к формированию ответа

### Шаг 11: Отправка email с кодом
- Вызов `send_verification_email(email, code)`
- Попытка отправить письмо с кодом подтверждения

### Шаг 12: Обработка результата отправки email
- **Если email отправлен успешно:**
  - `email_sent = True`
  - Проверка режима работы:
    - **Режим разработки:** `message = "Регистрация успешна! Проверьте консоль сервера для получения кода подтверждения."`
    - **Продакшн:** `message = "Регистрация успешна! Проверьте вашу почту - мы отправили код подтверждения."`
- **Если email не отправлен:**
  - `email_sent = False`
  - `message = "Регистрация успешна, но не удалось отправить письмо: {error}. Проверьте консоль сервера."`

### Шаг 13: Формирование ответа
- Создание объекта ответа:
  ```json
  {
    "message": "...",
    "email_sent": true/false,
    "email_message": "...",
    "user": {
      "user_id": ...,
      "email": ...,
      "first_name": ...,
      "last_name": ...,
      "phone_number": ...,
      "email_verified": false
    },
    "verification_code": "..." // только в режиме разработки
  }
  ```

### Шаг 14: Обработка результата на клиенте
- **Если успешно:**
  - Вывод сообщения: `setMessage(result.message)`
  - Если `email_sent == true`:
    - Ожидание 1.5 секунды
    - Закрытие модального окна: `onClose()`
    - Переход на страницу подтверждения: `navigate("/verify-code", { state: { email } })`
  - Если `email_sent == false`:
    - Сообщение остается на экране
- **Если ошибка:**
  - Вывод сообщения об ошибке: `setMessage(error.message)`

### Шаг 15: Завершение
- Сброс флага загрузки: `setIsLoading(false)`
- КОНЕЦ

## Сложность алгоритма

- **Временная сложность:** O(1) - константная, все операции выполняются за константное время
- **Пространственная сложность:** O(1) - константная память

## Особенности алгоритма

1. **Безопасность паролей:** Пароли хэшируются с использованием bcrypt перед сохранением
2. **Генерация кода:** Случайный 6-значный код для подтверждения email
3. **Роль по умолчанию:** Все новые пользователи получают роль `customer`
4. **Email не подтвержден:** При создании `email_verified = FALSE`
5. **Обработка ошибок:** Различные сообщения в зависимости от результата отправки email
6. **Режим разработки:** В режиме разработки код выводится в консоль сервера
7. **Автоматический переход:** После успешной регистрации пользователь переходит на страницу подтверждения

## Типы ошибок

### HTTP 400 (Bad Request)
- **Причина 1:** Не заполнены обязательные поля (email или password)
- **Сообщение:** "Email и пароль обязательны"
- **Причина 2:** Email уже используется
- **Сообщение:** Описание ошибки из базы данных

### Ошибка отправки email
- **Причина:** Проблемы с SMTP сервером или настройками
- **Результат:** `email_sent = False`, но регистрация успешна
- **Сообщение:** "Регистрация успешна, но не удалось отправить письмо"

## Примеры сценариев

### Сценарий 1: Успешная регистрация с отправкой email
- Все данные заполнены корректно
- Email уникален
- SMTP настроен правильно
- **Результат:** Пользователь создан, email отправлен, переход на страницу подтверждения

### Сценарий 2: Успешная регистрация в режиме разработки
- Все данные заполнены корректно
- Email уникален
- SMTP не настроен
- **Результат:** Пользователь создан, код выведен в консоль сервера, переход на страницу подтверждения

### Сценарий 3: Email уже используется
- Email уже существует в базе данных
- **Результат:** Ошибка 400 "Email уже используется"

### Сценарий 4: Пустые обязательные поля
- Email или password не заполнены
- **Результат:** Ошибка 400 "Email и пароль обязательны"

### Сценарий 5: Ошибка отправки email
- Пользователь создан успешно
- SMTP сервер недоступен
- **Результат:** Пользователь создан, но email не отправлен, сообщение об ошибке

## Безопасность

1. **Хэширование паролей:** Используется bcrypt с автоматической генерацией соли
2. **Уникальность email:** Проверка на уровне базы данных (UNIQUE constraint)
3. **Валидация данных:** Проверка обязательных полей на сервере
4. **Код подтверждения:** Случайный 6-значный код для предотвращения подбора
5. **Роль по умолчанию:** Новые пользователи получают минимальные права (customer)

## Следующие шаги после регистрации

После успешной регистрации пользователь должен:
1. Получить код подтверждения на email (или в консоль сервера в режиме разработки)
2. Перейти на страницу подтверждения email
3. Ввести полученный код
4. После подтверждения email сможет авторизоваться в системе
